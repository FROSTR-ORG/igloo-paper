Tooltips & Help Text — Contextual Help
Contextual Help Text
Signer status
The signer must be running to handle signature requests from clients. When active, it will communicate with other nodes through your configured relays.
Relay configuration
Part of your device profile. You must be connected to at least one relay to communicate with other signers. Each peer configures their own relay list.
Import Credentials (bfonboard / bfprofile)
Use an onboarding package (bfonboard1...) to onboard a device, or a profile backup (bfprofile1...) to import a saved profile.
Profile password
Used to encrypt/decrypt a profile on this device.
Recover NSEC
Use the threshold number of shares to recover the original nsec.
Peer pool & readiness
Each peer pre-exchanges signing capacity with the group. Rings, bars, and counts show whether that peer is ready to co-sign right now. Blue = healthy, amber = low, red = critical, gray = offline. The ~ready pill is the approximate total signatures currently available across online peers. Capacity refills automatically during sync and periodic health checks.
Sync events
Teal SYNC entries in the event log mean signing capacity was just replenished with a peer. This happens automatically during health checks. If you see frequent SYNC events, it means the system is actively signing and healthy.
Permission tags
Assign SIGN, ECDH, PING, and ONBOARD tags to peers you trust for automatic peer requests. Requests not explicitly allowed by signer policies require a peer policy decision.
Peer policy decisions
When an external request is not explicitly allowed, you will see a peer policy decision prompt with method details and requester pubkey. Verify details before allowing.
Request timeout
Each request has a timeout. If no response is provided before timeout, the request is rejected and the client is notified.
Group profile (read-only)
Shared across all peers, synced via Nostr. Cannot be edited from a single device — changes require group consensus.
Device profile
Controls how this specific share operates — relay list, peer policies, and local preferences. Changes only affect this device.
Profile Backup
A full encrypted backup of a saved profile. Import it on another device or keep it as an encrypted archive.
Export Profile
Create an encrypted bfprofile1... backup to import this profile on another device.
Export Share
Copy the raw share key in hex format. This export is unencrypted, so handle it with care.
Source share intake
Paste threshold-worth of bfshare packages and their passwords. Each share is decrypted and validated against the current group configuration. Old devices do not need to be online — only bfshare material and passwords are required.
Target assignment
Each rotated member slot is assigned an intent. Exactly one must be Local Replace (replaces the current device). Others are marked as New Device or Rotate Existing Device — this changes labeling and instructions only, not the package format. All remote outputs are bfonboard packages.
Distribution password
Encrypts the bfonboard packages emitted for remote devices. Each recipient will need this password to decrypt and adopt their rotated share. Set during the Review & Generate step.
Identity changes after rotation
After rotation, each device receives a new share public key and a new profile ID (derived from the share key). The group public key (npub) remains unchanged — this is the stable identity anchor. Device labels, relay configurations, and peer permissions are preserved.
Group name (read-only during rotation)
The group name is inherited and read-only during share rotation. Renaming is outside the rotation contract. To change the group name, use keyset settings separately.
